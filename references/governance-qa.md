# QA、数据质量与治理

埋点体系的死法不是「没数据」，而是「有错的数据但没人发现，团队还拿它做了决策」。本文件是工作流程第 7 步的细则——让这套体系活得久、可信赖。

## 1. 落地流程

标准四步，每步有明确责任人：
1. **产品/数据提需求**：在 Tracking Plan 里写清事件、属性、触发时机、对应指标、owner。
2. **研发埋点**：按方案实现，统一走 analytics 封装层注入公共属性。
3. **测试验证**：在真机/测试环境核对事件是否上报、属性是否齐全、类型是否正确、口径是否符合预期。
4. **数据分析使用**：进入报表/看板，被指标真正消费。

关键：**埋点验收是发布流程的一环**，不是上线后才补。没验证过的埋点等于没埋。

## 2. 上线前 QA 清单

逐事件核对：
- [ ] 事件名符合命名规范（大小写/时态/词表/object-action），是固定字符串、无近似重名。
- [ ] 必填属性全部存在；属性类型与 Tracking Plan 声明一致（数值就是数值）。
- [ ] 属性值在合理范围/枚举内（如 `payment_type` 只出现约定的几个值）。
- [ ] 公共属性（env、app_version、platform、distinct_id）已由封装层自动注入。
- [ ] 触发时机正确：该触发时触发、不该触发时不触发、不重复触发（去抖）。
- [ ] **identity stitching**：首屏初始化一次、登录后 `identify()` 一次；登录前后行为能拼成同一人（专项验证，这类错误不报错）。
- [ ] 错误/失败态也被采集（如 `payment_failed`），否则漏斗分母失真、问题不可见。
- [ ] 关键转化事件已做客户端/服务端交叉校验（如双采，确认无重复计数）。

## 3. 上线后数据质量监控

- **单一封装层**：所有上报走同一个 wrapper，集中注入默认值与做基本校验，避免各端各写、口径漂移。
- **量级告警**：核心事件设阈值告警——必填属性缺失率突增、日量级骤升骤降、新值涌入枚举属性，都应触发预警。
- **定期审计**：如每季度对 Top 事件做一次体检，看量级完整性、是否有异常、是否有已废弃却仍在上报的事件。
- **schema 校验**：有条件的话在数据入口（如 Segment Protocols / 自建网关）按 JSON Schema 校验，对不合规事件采取放行+记录 / 删属性 / 拦截 三种策略之一。

## 4. 事件生命周期

每个事件有状态机：
```
proposed（提案） → active（在用） → deprecated（弃用） → removed（移除）
```

变更规则：
- **改名 = 迁移，不是裸改名**。裸改名会把一条时间序列劈成两段断裂的曲线。正确做法：新旧名**并行上报一个发布周期**，下游切换到新名后再下线旧名。
- 任何破坏性变更（删属性、改类型、改口径）都要走评审，并发布**变更说明 / release notes**，让用看板的人知道何时该更新口径。
- 限制谁能新建事件/属性（设 data steward / gatekeeper），保护词表与命名空间不被随意扩张。

## 5. 治理机制：把埋点当代码

「活在电子表格里的 Tracking Plan 一定会跟现实脱节；被写进 CI、上线前就校验的方案才扛得住时间。」

- **版本化**：把事件 schema 文件随代码一起进 Git，和业务代码同生命周期。
- **CI 校验**：在合并前自动校验新埋点是否符合命名规范、必填项、类型；不合规直接 block。
- **代码评审四关**（每个埋点变更过一遍）：
  1. 命名关：大小写/格式/时态/词表是否合规，是否固定字符串、无近似重名。
  2. 属性与类型关：必填齐全、类型不变、不超属性上限。
  3. 归属关：每个事件有明确 owner（没有 owner 的事件 = owner 一离职就腐烂的隐患）。
  4. 弃用关：破坏性变更是否带了迁移方案而不是裸改。
- **代码生成**：用平台工具（Amplitude Data、PostHog CLI、Avo 等）从 schema 生成带类型的埋点代码，减少手写错误。
- 本 skill 提供 `scripts/validate_tracking_plan.py`，可对填好的 Tracking Plan CSV 做命名/必填/类型一致性的离线体检，作为 CI 的最小起点。

## 6. 常见腐烂模式与对策

| 腐烂模式 | 根因 | 对策 |
|---|---|---|
| 命名漂移 | 各研发各起名 | 受控词表 + CI 命名关 |
| 属性类型突变 | 同名属性时而数字时而字符串 | 入口 schema 校验、CI 类型关 |
| 全埋点失效 | UI 改版后自动采点对不上 | 核心 KPI 用代码埋点，不依赖自动采集 |
| 身份分裂 | 登录前后 distinct_id 不一致 | 首屏 + 登录后各 `identify()` 一次 |
| 无人维护 | 事件没 owner | 强制 owner 字段 + 归属关评审 |
| 客户端丢数 | 广告/隐私拦截吞掉 25–40% | 关键事件服务端采集/双采 |
| 指标吵架 | 口径没写清 | 每个指标定义分子/分母/时间窗/去重/过滤 |
