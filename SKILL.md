# 用户行为监测 Skill（埋点体系设计 / Tracking Plan Designer）

帮助用户把「我想看用户数据」这种模糊诉求，变成一套**目标对齐、口径清晰、能落地、可维护**的用户行为数据监测体系，最终交付一份可直接给研发执行的 **Tracking Plan（埋点方案 / 事件字典）**。

## 核心心法：先想清楚「为什么」，再决定「埋什么」

最常见的失败模式是**先埋点、再想用途**——结果埋点泛滥、口径混乱、没人维护、最后没人用。本 skill 强制反过来走：

```
业务目标 (North Star)
  → 用户旅程 (UJM)
    → 指标 (OSM / HEART)
      → 事件 (Event)
        → 埋点方案 (Tracking Plan) + 属性 (Property)
          → 研发落地 → QA 验证 → 数据治理
```

判定一个事件该不该埋，只问两句话：
- **算不出来的指标是空的**：每个指标必须能由某几个事件 + 属性算出来。
- **采不到指标的事件是冗余的**：埋了但喂不进任何指标的事件，就是噪声，不埋。

## 底层数据模型：Event–Property–User

几乎所有主流分析平台（神策、GrowingIO、数数、Amplitude、Mixpanel、GA4）都建立在同一抽象上——**事件模型**：

- **User（用户）**：谁做的 —— 用户 ID + 用户属性（新老、来源渠道、会员等级、城市…）。
- **Event（事件）**：做了什么 —— 曝光、点击、生成、采纳、支付、转化…
- **Property（属性）**：在什么上下文下做的 —— 物料 ID、坑位、页面、来源、设备、金额…

这三要素一旦定清楚，后面的**漏斗、留存、归因、分群**都只是在它之上做切片。模型定对，比堆功能重要得多。深入细节（用户标识 identify、`$set` vs `$set_once`、客户端 vs 服务端采集、事件 vs 属性的取舍）见 `references/event-model.md`。

## 工作流程（按这个顺序推进，不要跳步）

根据用户当前所处的阶段切入；如果对方一上来就让你「列一堆要埋的事件」，先把他拉回第 1–3 步对齐目标，否则产出物必然过拟合、不可用。

### 第 1 步：锁定业务目标 / 北极星指标
先问清楚：**这套监测体系到底服务于哪个决策？** 没有这个，后面全是无根之木。
- 帮用户找到 **North Star Metric（北极星指标）**：能代表用户从产品中获得核心价值、且与长期商业成功正相关的那一个数。
- 区分北极星与虚荣指标（PV、注册数往往是虚荣指标）。
- 框架细节（North Star / AARRR）见 `references/methodology.md`。

### 第 2 步：拆解用户旅程（UJM, User Journey Map）
把目标落到用户的关键路径上：
- 列出用户从进入到达成目标的关键阶段（如：曝光 → 进入 → 体验核心功能 → 激活 → 转化 → 复购）。
- 每一步标出**关键动作**和**潜在流失点**。
- 这一步决定了「值得监测的瞬间」在哪里——后面的事件主要就埋在这些关键动作与流失点上。

### 第 3 步：用 OSM / HEART 把目标翻译成指标
- **OSM**（Objective–Strategy–Measurement，目标–策略–度量）：把每个业务目标拆成「采取什么策略」和「用什么可量化的数来衡量」。
- 体验类 / 功能类产品可叠加 **HEART**（Happiness、Engagement、Adoption、Retention、Task success）。
- 给每个指标写清楚**精确口径**：分子、分母、时间窗、去重方式、过滤条件。口径不清的指标，到了报表上一定会吵架。
- 框架与示例见 `references/methodology.md`。

### 第 4 步：由指标反推事件（这是承上启下的关键一步）
对每个指标，反问「要算它，需要捕捉用户的哪几个动作？」，由此得出事件清单。
- 一个事件 = 一个**离散的、有分析价值的**用户动作。
- 控制规模：一份 Tracking Plan 通常 **10–200 个事件**，每个事件 ≤ 约 20 个属性。少于 10 个难做漏斗，多于 200 个没人维护。
- 「多个事件」还是「一个事件 + 属性区分」？**优先用属性区分**（例如统一一个 `order_completed` 事件，用 `payment_type` 属性区分支付方式，而不是为每种支付方式建一个事件）——这样做漏斗、做对比时口径统一。判断细节见 `references/event-model.md`。

### 第 5 步：设计属性与命名规范，产出 Tracking Plan
这是核心交付物。**命名规范必须在写第一个事件之前就定好**，否则后期一定漂移。
- 采用 **object-action（对象–动作）** 命名：`product_viewed`、`button_clicked`、`order_completed`。
- 大小写、时态、词表三轴各选一种并全程统一（平台会把 `Song Played` 和 `song played` 当成两个事件）。
- **事件名和属性名必须是代码里的固定字符串，绝不能动态拼接**（绝不要把用户 ID、商品名拼进事件名——那会制造无法治理的无限命名空间）。变量值只能放进属性值里。
- 完整命名规范（动词受控词表、属性命名 `object_adjective`、布尔 `is_`/`has_`、时间戳 `_timestamp`、GA4/各平台硬限制）见 `references/naming-conventions.md`。
- 用 `assets/tracking-plan-template.csv` 作为交付模板；每个事件至少写清：事件名、中文释义、触发时机、所属页面/端、属性列表（名/类型/示例/是否必填）、对应指标、负责人、状态、版本。

### 第 6 步：决定怎么采集（埋点方式）
- **代码埋点（手动）**：精准、可带丰富业务属性，适合核心转化与关键业务事件——成本高。
- **全埋点 / 无埋点（自动采集）**：SDK 自动采点击/浏览，覆盖广、上线快，但属性贫、口径粗、UI 改动易失效。
- **可视化埋点**：在后台圈选元素配置，介于两者之间。
- **推荐：混合模式** —— 全埋点打底拿基础点击流，核心 KPI（注册、支付、收入）用代码埋点精细采集。对收入等关键事件，可在**客户端 + 服务端双采**（用不同事件名以免重复计数），因为客户端事件会被广告拦截/隐私拦截吞掉一部分（业内估计 25–40%）。细节见 `references/event-model.md`。

### 第 7 步：QA、验证与数据治理（决定这套体系能活多久）
埋点最大的风险不是没数据，而是**有错的数据却没人发现**。
- 上线前：用单一 analytics 封装层统一注入默认属性（环境、版本号），并在测试 / CI 中校验必填属性与类型。
- 上线后：核心事件设量级告警（属性缺失、量级突变），定期（如季度）做埋点审计。
- 生命周期：每个事件有 owner、有状态（proposed → active → deprecated → removed）；改名走「新旧名并行一个版本」的迁移，绝不裸改名（裸改名 = 制造两段断裂的时间序列）。
- 可用 `scripts/validate_tracking_plan.py` 对填好的 Tracking Plan CSV 做命名规范 / 必填项 / 类型一致性的自动体检。
- 完整 QA 清单与治理机制见 `references/governance-qa.md`。

## 交付物（默认产出）

除非用户另有要求，最终交付应包含：
1. **目标—旅程—指标—事件 对齐表**：一眼能看出每个事件服务于哪个指标、哪个目标（保证无冗余、无遗漏）。
2. **Tracking Plan / 事件字典**：基于 `assets/tracking-plan-template.csv` 填好的完整方案。
3. **命名规范与埋点约定**：本项目采用的大小写/时态/词表，以及公共属性（super properties）。
4. **采集方式建议 + QA / 治理机制**：每类事件用什么方式埋，如何验证与维护。

## 关键原则速记
- 从「为什么」开始，不从「埋什么」开始。
- 指标先有精确口径，事件由指标反推。
- 命名规范先行；事件名固定、变量进属性。
- 优先「一个事件 + 属性」而非「拆成多个事件」。
- 混合采集；核心转化代码埋点 + 关键事件双端校验。
- 把埋点当代码来治理：有 owner、有 schema、有迁移、有 CI 校验。
